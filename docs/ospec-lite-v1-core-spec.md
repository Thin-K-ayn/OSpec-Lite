# OSpec Lite V1 Core Spec

## Purpose

This document defines the core V1 implementation of `ospec-lite`.

V1 core is intentionally generic:

- no stack-specific logic hard-coded into the scanner
- no plugin runtime
- no queue runner
- no destructive doc regeneration after init

Its only job is to make a repository easier for coding agents to understand and easier for humans to track lightweight changes.

## Primary Goal

After `oslite init .`, a team member should be able to open either Codex or Claude Code and get better results immediately because the repo now contains:

- clear top-level agent instructions
- a small project knowledge layer
- a machine-readable project summary
- an optional repo-local authoring pack
- a simple change workflow

## Built-In Agent Targets

V1 core explicitly supports these two agent targets:

- `codex`
- `claude-code`

This support belongs in the core because these tools are the primary consumers of the generated repo knowledge.

## Optional Profiles

V1 core may optionally initialize a repository with an asset-based profile.

Rules:

- profiles live under `profiles/<id>/`
- profiles are content packs, not executable plugins
- profiles may override the initial markdown templates for a selected repo
- profiles may add a shared authoring pack under `docs/agents/authoring/`
- profiles may declare required repo paths that must exist before init can proceed
- profiles may define profile-specific init questions and bootstrap defaults
- profiles should encode repository-reading workflows, not project-private facts
- profiles do not change the generic scanner or add direct model orchestration

## Repo-Local Plugin Assets

V1 may manage repo-local Codex companion plugins as assets without introducing a plugin runtime.

Rules:

- bundled plugin assets may live under `plugins/<id>/`
- a repo-local marketplace may live at `.agents/plugins/marketplace.json`
- plugin commands may scaffold new local plugins or copy existing local plugin directories
- plugin commands may install bundled defaults into a target repository
- V1 does not execute plugin code, resolve remote marketplaces, or add provider-specific orchestration

## One-Time Init Behavior

`oslite init .` is a one-time bootstrap in V1.

Rules:

- if the repo is uninitialized, create the full knowledge layer
- if the repo is already initialized, print an informative log and exit without mutating files
- if initialization is incomplete, print an incomplete-initialization message and stop

## Refresh Behavior

`oslite refresh .` is the non-destructive maintenance command in V1.

Rules:

- refresh requires repo state `initialized`
- refresh rescans the repository and rewrites only machine-managed artifacts
- refresh updates `.oslite/index.json`
- refresh may rewrite the managed sections inside `AGENTS.md` and `CLAUDE.md`
- refresh must not overwrite human-owned project docs or authoring-pack docs
- refresh should report human-owned docs whose generated suggestions have drifted
- refresh should initialize suggestion hashes on older repos that do not have them yet

Required initialization markers:

- `.oslite/config.json`
- `.oslite/index.json`
- `AGENTS.md`
- `CLAUDE.md`
- `docs/project/overview.md`
- `docs/project/architecture.md`
- `docs/project/repo-map.md`
- `docs/project/coding-rules.md`
- `docs/project/glossary.md`
- `docs/project/entrypoints.md`
- `docs/agents/quickstart.md`
- `docs/agents/change-playbook.md`
- `changes/active/`
- `changes/archived/`

Example already-initialized log:


## Ownership Model

### Tool-Owned

- `.oslite/config.json`
- `.oslite/index.json`
- managed sections inside `AGENTS.md` and `CLAUDE.md`

These are generated by the tool and may be treated as machine-managed.

### Human-Owned After Init

- `AGENTS.md`
- `CLAUDE.md`
- `docs/project/*.md`
- `docs/agents/*.md`

The tool creates the initial version once. After that, V1 does not try to rewrite or merge these files.

## Agent Entry Files

### `AGENTS.md`

Role:

- primary entrypoint for Codex-style environments
- concise human-plus-agent instructions
- links to deeper docs

Required sections:

- What This Repo Is
- Hard Rules
- High-Risk Areas
- Preferred Commands
- How To Verify
- Generated Files
- Ask-Before-Edit Areas
- Read Next

Content rules:

- keep it short
- include enough direct guidance that an agent can act even before opening deeper docs
- link to `docs/project/*` and `docs/agents/change-playbook.md`

### `CLAUDE.md`

Role:

- primary project memory file for Claude Code
- should avoid duplicating all shared guidance

Required sections:

- Shared Instructions Import
- Claude-Specific Notes

Recommended content shape:

```md
# Claude Code Project Memory

@AGENTS.md

## Claude Code Notes

- Prefer the shared project rules above.
- Use `docs/agents/change-playbook.md` for the change workflow.
```

Rules:

- import `AGENTS.md` to avoid duplicated project guidance
- keep Claude-specific instructions minimal
- do not fork project rules between `AGENTS.md` and `CLAUDE.md`

```json
  "agentDocsRoot": "docs/agents",

```

Notes:

- do not store a repo-level `ready` flag
- V1 should store facts about initialization, not opinionated maturity states

## `.oslite/index.json`

This is the machine-readable summary for tools and agents.

Suggested shape:

```json
{
  "version": 1,
  "generatedAt": "2026-04-08T10:12:00+08:00",
  "project": {
    "name": "repo-name",
    "documentLanguage": "zh-CN"
  },
  "agentTargets": ["codex", "claude-code"],
  "roots": {
    "repoRoot": ".",
    "changeRoot": "changes"
  },
  "directoryMap": [],
  "entrypoints": [],
  "rules": [],
  "importantFiles": [],
  "glossarySeeds": [],
  "signals": {}
}
```

### Required Index Sections

`directoryMap`

- top-level directory inventory
- optional one-line inferred role when confidence is reasonable

`entrypoints`

- likely startup, bootstrap, or central orchestration files

`rules`

- distilled machine-readable editing rules

`importantFiles`

- files an agent should inspect early

`glossarySeeds`

- terms inferred from folder and file names that humans may later refine

`signals`

- basic scanner observations such as `hasPackageJson`, `hasGit`, `hasUnityProjectSettings`, `hasScriptDir`

## Generic Scanner Responsibilities

The V1 core scanner should be generic and conservative.

It should extract:

1. top-level directory inventory
2. common manifest and config files
3. likely entrypoint files
4. obvious agent-relevant rules from known instruction files
5. candidate glossary terms from directory and file names

### 1. Top-Level Directory Inventory

Collect top-level directories and classify them when confidence is high.

Examples of useful generic categories:

- source
- docs
- tests
- assets
- tools
- config
- unknown

### 2. Common Manifest And Config Files

Detect files such as:

- `package.json`
- `README.md`
- `AGENTS.md`
- `CLAUDE.md`
- `.gitignore`
- `*.sln`
- `*.csproj`
- `ProjectSettings/`
- `Assets/`

These should influence:

- `overview.md`
- `architecture.md`
- `.oslite/index.json.signals`

### 3. Likely Entrypoint Files

Use generic filename and centrality heuristics.

Examples:

- `main.*`
- `index.*`
- `app.*`
- `Program.cs`
- root-level bootstrap files
- files with many imports or references

V1 should be explicit that these are inferred entrypoints, not guaranteed truth.

### 4. Rule Harvesting

If existing repo instruction files are present, the scanner may harvest high-signal rules from:

- `AGENTS.md`
- `CLAUDE.md`
- repo-local docs explicitly about conventions

This should stay conservative:

- do not paraphrase aggressively
- prefer short distilled rule bullets in `.oslite/index.json.rules`

### 5. Glossary Seeds

Extract candidate domain terms from:

- top-level folder names
- frequently repeated file prefixes
- obvious product feature names

These go into `glossary.md` as seeds, not confirmed truth.

## Generated Markdown Contracts

### `docs/project/overview.md`

Purpose:

- explain what the repository appears to be

Required sections:

- Project Summary
- Repo Signals
- Main Working Areas
- Needs Human Confirmation

### `docs/project/architecture.md`

Purpose:

- explain the high-level structure in a way agents can reason about

Required sections:

- Runtime Shape
- Important Boundaries
- Central Orchestration Points
- Risks And Unknowns

### `docs/project/repo-map.md`

Purpose:

- tell agents where major things live

Required sections:

- Top-Level Directories
- High-Value Files
- Where To Add New Work
- Needs Human Confirmation

### `docs/project/coding-rules.md`

Purpose:

- hold the repo's hard and soft editing rules

Required sections:

- Hard Rules
- Recommended Practices
- Compatibility Expectations
- Documentation Expectations

### `docs/project/glossary.md`

Purpose:

- give agents a stable place for project vocabulary

Required sections:

- Confirmed Terms
- Candidate Terms From Scan

### `docs/project/entrypoints.md`

Purpose:

- surface the files that likely drive startup or central behavior

Required sections:

- Likely Entrypoints
- High-Risk Central Files
- Notes For Safe Investigation

### `docs/agents/quickstart.md`

Purpose:

- help an agent orient quickly

Required sections:

- Read These First
- How To Explore Safely
- How To Choose Where Code Belongs
- How To Report Work
- What To Double-Check Before Editing

### `docs/agents/change-playbook.md`

Purpose:

- define the lightweight change workflow

Required sections:

- Start A Change
- Plan Before Editing
- Record Applied Work
- Record Verification
- Archive When Done

### `docs/agents/authoring/*`

Purpose:

- hold the shared repo-local prompt pack used by both Codex and Claude Code

Required files:

- `doc-contract.md`
- `project-brief.md`
- `repo-reading-checklist.md`
- `evidence-map.md`
- `fill-project-docs.md`
- `doc-task-checklist.json`

Rules:

- `doc-task-checklist.json` is the source of truth for documentation verification
- `fill-project-docs.md` must prescribe a two-stage flow: collect evidence first, then fill final docs
- `evidence-map.md` is the repo-local intermediate artifact that captures the evidence set before final docs are rewritten
- agent-specific guidance should point to this pack instead of duplicating it
- `unity-tolua-game` may prefill `project-brief.md` from init answers such as `projectName` and `bootstrapAgent`

### Thin Agent Wrappers

Profiled repositories may also generate thin agent wrappers that only point to the repo-local authoring pack.

Examples:

- Codex wrapper skill: `.codex/skills/oslite-fill-project-docs/SKILL.md`
- Claude Code wrapper command: `.claude/commands/oslite-fill-project-docs.md`

Rules:

- wrappers should not define a second workflow
- wrappers should point back to `docs/agents/authoring/fill-project-docs.md`
- wrappers should remind the agent to fill `evidence-map.md` before final docs
- wrappers should end by telling the agent to run `oslite docs verify .` when available

## Minimal Change Flow

A V1 change is a lightweight, repo-local work record for one non-trivial task.

It is intentionally not:

- a replacement for git branches
- a replacement for commits
- a heavyweight issue tracker

Its job is to keep request, plan, implementation notes, and verification close to the code so both humans and coding agents can review or resume work safely.

Each active change lives under:

```text
changes/active/<slug>/
  change.json
  request.md
  plan.md
  apply.md
  verify.md
```

Recommended semantics:

- `request.md`: what was asked for, the intended scope, and acceptance notes
- `plan.md`: intended approach, expected files/modules, and known risks before editing
- `apply.md`: what was actually changed and how it differed from plan
- `verify.md`: commands, results, manual validation, and remaining risks
- `change.json`: machine-readable state and metadata for tooling

Recommended use cases:

- changes that touch multiple files
- changes that alter behavior, interfaces, or architectural understanding
- changes that need explicit validation notes
- changes likely to span multiple sessions or require handoff

Transition requirements:

- `change apply` should reject empty `affects`, template placeholders, or missing applied-work detail
- `change verify` should re-check the apply requirements and also reject missing command/result evidence

Allowed statuses:

- `draft`
- `active`
- `applied`
- `verified`
- `archived`

## V1 `status` Output

`oslite status .` should answer only these questions:

1. is the repo initialized?
2. which agent targets are configured?
3. where are the primary docs?
4. is there an active change?

Suggested output:

```text
OSpec Lite Status
Initialized: yes
State: initialized
Project: repo-name
Profile: unity-tolua-game
Bootstrap agent: codex

- codex: AGENTS.md

- claude-code: .claude/commands/oslite-fill-project-docs.md

```

## `docs verify`

`oslite docs verify .` is a deterministic verifier for profile-driven documentation.

Rules:

- load `docs/agents/authoring/doc-task-checklist.json`
- fail on missing authoring-pack files, missing headings, placeholder text, missing evidence blocks, missing status markers, missing evidence paths, and forbidden scope expansion
- allow profile-specific rules such as section-level required snippets
- print the active profile id and enumerated violations on failure

## Recommended Build Order

1. init guards and bootstrap
2. generic repo scanner
3. Markdown template renderer
4. `AGENTS.md` generation
5. `CLAUDE.md` generation
6. `.oslite/index.json` generation
7. `status`
8. `change new`
9. `change archive`

## Out Of Scope For V1

- doc regeneration after init
- queue execution
- plugin execution runtimes
- remote plugin marketplace sync
- Jira or Excel import
- merge logic for human-edited knowledge docs

V1 includes asset-based profiles, but not executable profile plugins or provider-managed doc generation.
The `unity-tolua-game` profile may hard-code `Script/MJGame.lua` as the main entry anchor, but other project facts must come from the repo-local evidence workflow.
